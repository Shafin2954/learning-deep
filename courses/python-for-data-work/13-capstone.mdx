## The project

Build a **columnar analytical engine from scratch using only NumPy**. It must handle
multi-million-row data under a fixed time and memory budget. The goal is to prove that you
understand what pandas and polars are doing internally â€” not just how to call them.

You will implement three operations that together cover the core of any groupby-based
analytical query: a **hash groupby-aggregate**, a **hash join**, and a **vectorized filter**.
Then you will run your engine against a messy real-world-style CSV and verify correctness
against a pandas reference.

## Hard constraints (no vibe coding)

- **No AI writes your logic or your design.** Use only the docs and these chapters.
- **No `pandas.groupby`, `pandas.merge`, `DataFrame.join`, or `DataFrame.merge`** for the
  core engine. You may use pandas only for reading the CSV and for correctness comparison.
- **No polars for the engine.** The point is to build it.
- **NumPy only** for the engine internals: `np.unique`, `np.argsort`, `np.where`, fancy
  indexing, boolean masks, ufuncs, and `np.einsum` are all fair game.
- Every design decision must trace to a measurement (time or memory) â€” not vibes.

## What it must contain (and the chapter each tests)

### 1. Vectorized filter (ch. 1â€“3)
A function `filter_rows(table, col_idx, condition)` that returns a boolean mask over a
NumPy 2-D array using only ufuncs and boolean indexing. No Python loop per row.

Conditions to support: `>`, `<`, `==`, `!=`, `between(lo, hi)`. They must compose with
`&` and `|` (ch. 3).

### 2. Hash groupby-aggregate (ch. 1â€“3)
A function `groupby_agg(data, key_col, val_col, aggfunc)` that:
1. Encodes the group key column into integer codes using `np.unique(..., return_inverse=True)`.
2. Sorts by code using `np.argsort` (for cache-friendly access).
3. Computes the aggregate (`sum`, `mean`, `count`, `max`, `min`) over each group using only
   NumPy: `np.add.reduceat` or `np.bincount` + fancy indexing are the right tools.
4. Returns a structured result: `(unique_keys, agg_values)`.

Benchmark: must be within 5Ă— of `pandas.groupby.sum()` on 10 million rows.

### 3. Hash join (ch. 1â€“3, ch. 7)
A function `hash_join(left, right, left_key_col, right_key_col)` that:
1. Builds a Python dict (or uses `np.searchsorted` on sorted keys) to map right-side keys
   to their row indices.
2. For each left-side key, looks up matching right-side rows.
3. Returns the row indices to use with fancy indexing to form the joined table.

Implement at least an **inner join**. A left join (NaN fill for non-matches) for extra
credit.

## Acceptance criteria

- **Correctness:** your engine's output matches `pandas.groupby.sum()` and `pandas.merge()`
  to within float32 tolerance on the same data.
- **Performance:** the groupby-aggregate on 10 million rows completes in under 3 seconds on
  a standard laptop (no GPU).
- **No leakage:** the engine uses zero pandas/polars for the core â€” confirmed by removing the
  imports and verifying it still runs.
- **Explanation:** you can explain why `np.add.reduceat` is faster than a Python loop over
  groups, and why the sort step improves cache performance.
