## Why Standard Attention Doesn't Scale

Self-attention computes a score between every pair of tokens in the sequence. For a sequence of length $n$, that's $n^2$ pairs — and you need to store the full $n \times n$ attention matrix. Both time and memory scale as $O(n^2)$.

For $n = 1{,}000$: 1M entries. For $n = 128{,}000$ (a 128K context window): 16.4 billion entries per layer. At float16 (2 bytes each), that's 33 GB just for the attention matrix in one layer. GPT-4-class models have ~96 layers.

$$
\text{Attention}(Q, K, V) = \text{softmax}\!\left(\frac{QK^T}{\sqrt{d_k}}\right)V
$$

The bottleneck is $QK^T$ — an $(n \times d_k) \cdot (d_k \times n)$ matmul that produces an $(n \times n)$ matrix.

```python
import numpy as np

def attention_memory_bytes(n, d_k, num_heads, num_layers, batch=1, dtype_bytes=2):
    """Memory for the attention score matrices (QK^T) across all heads and layers."""
    # Each head: n x n matrix
    per_head = n * n * dtype_bytes
    total = per_head * num_heads * num_layers * batch
    return total

# GPT-2 small: 12 heads, 12 layers, d_k=64
for seq_len in [512, 2048, 8192, 32768]:
    mem = attention_memory_bytes(seq_len, 64, 12, 12)
    print(f"n={seq_len:6d}: {mem / 1e9:.2f} GB")
```

```output
n=   512: 0.01 GB
n=  2048: 0.15 GB
n=  8192: 2.42 GB
n= 32768: 38.65 GB
```

The jump from 2K to 32K context is 256×. This is why long-context models need architectural changes, not just more GPU memory.

## The KV Cache Problem

At inference time, attention has a different bottleneck: the **KV cache**. When generating token by token, you don't recompute the full attention from scratch each step — you cache the keys and values for all past tokens.

For each new token, the model:
1. Computes Q, K, V for the new token
2. Appends K and V to the cache
3. Attends new Q against all cached K and V

The KV cache size at step $t$ (sequence length $t$):

$$
\text{KV cache} = 2 \times t \times n_{\text{layers}} \times n_{\text{heads}} \times d_{\text{head}} \times \text{dtype bytes}
$$

```python
def kv_cache_gb(seq_len, num_layers, num_heads, d_head, batch=1, dtype_bytes=2):
    """KV cache memory in GB."""
    # 2 = keys + values
    total_bytes = 2 * seq_len * num_layers * num_heads * d_head * batch * dtype_bytes
    return total_bytes / 1e9

# Llama-3 8B: 32 layers, 8 KV heads (GQA), d_head=128
for ctx in [4096, 32768, 128000]:
    gb = kv_cache_gb(ctx, num_layers=32, num_heads=8, d_head=128, batch=1)
    print(f"ctx={ctx:7d}: {gb:.2f} GB KV cache")

# Compare: batch=32
print()
for ctx in [4096, 32768]:
    gb = kv_cache_gb(ctx, 32, 8, 128, batch=32)
    print(f"ctx={ctx:7d}, batch=32: {gb:.2f} GB")
```

```output
ctx=   4096: 0.27 GB KV cache
ctx=  32768: 2.15 GB KV cache
ctx= 128000: 8.39 GB KV cache

ctx=   4096, batch=32: 8.59 GB
ctx=  32768, batch=32: 68.72 GB
```

Even with GQA (Grouped Query Attention) reducing KV heads from 32 to 8, a single 128K-context request uses 8.4 GB. Serving 32 concurrent users at 32K context uses 69 GB — more than the model weights themselves.

## Compute vs Memory Bandwidth

Modern GPUs have far more raw FLOPs than they can feed from memory. An A100 has 312 TFLOPS (BF16) but only 2 TB/s memory bandwidth. The ratio is ~156 FLOPs per byte transferred.

For attention at long contexts, you spend most time **reading and writing the KV cache** — a memory-bandwidth-bound operation. Adding more FLOPs doesn't help.

```python
# Roofline model: is an operation compute-bound or memory-bound?
a100_flops = 312e12   # FLOPs/sec BF16
a100_bw    = 2e12     # bytes/sec

arithmetic_intensity_break_even = a100_flops / a100_bw
print(f"Break-even arithmetic intensity: {arithmetic_intensity_break_even:.0f} FLOPs/byte")

# Attention at n=8192, d=128, one layer
n, d = 8192, 128
flops_qkv = 3 * n * d * d * 2   # three projections
flops_attn = n * n * d * 2       # QK^T and AV
bytes_read = (n * d * 3 + n * n) * 2  # Q, K, V, attn matrix in fp16
intensity = (flops_qkv + flops_attn) / bytes_read
print(f"Attention intensity at n=8192: {intensity:.1f} FLOPs/byte")
print(f"Memory-bound: {intensity < arithmetic_intensity_break_even}")
```

```output
Break-even arithmetic intensity: 156 FLOPs/byte
Attention intensity at n=8192: 63.5 FLOPs/byte
Memory-bound: True
```

Attention at long sequences is memory-bandwidth limited. This is why FlashAttention (ch. 12's Under the Trench) matters — it doesn't reduce FLOPs, it reduces memory reads/writes via tiling.

## The Alternatives

Three families of architectures attack this problem:

| Approach | How | Training | Inference | Quality |
|----------|-----|----------|-----------|---------|
| **Linear attention** | Kernel replaces softmax → $O(n)$ | Parallel | Recurrent $O(1)$/step | Weaker |
| **State space models** | Convolution train, recurrence infer | Parallel | $O(1)$/step | Near-attention |
| **Sparse attention** | Attend only to local/selected tokens | Parallel | Partial KV cache | Good |
| **MoE** | Sparse FFN, not sparse attention | Parallel | Parallel | Matches dense |

The next chapters cover each in depth.

## Try it Yourself

```python
import numpy as np

def compare_costs(seq_lengths, d_model=512, num_heads=8, num_layers=6):
    d_k = d_model // num_heads
    print(f"{'n':>8} | {'Attn QK^T (GB)':>16} | {'SSM state (MB)':>16} | {'Ratio':>8}")
    print("-" * 60)
    for n in seq_lengths:
        # Attention: QK^T matrix per head per layer
        attn_gb = n * n * num_heads * num_layers * 2 / 1e9
        # SSM: state size is O(n) not O(n^2) — fixed state dim D
        D = 16  # state dimension
        ssm_mb = n * D * num_layers * 2 / 1e6
        ratio = (n * n * num_heads) / (n * D)
        print(f"{n:>8} | {attn_gb:>16.3f} | {ssm_mb:>16.2f} | {ratio:>8.0f}x")

compare_costs([512, 2048, 8192, 32768, 131072])
```

    8192 |           0.805 |           1.44 |   49152x

## Exercise

The KV cache for a model with 40 layers, 40 KV heads (GQA-off), and $d_{\text{head}} = 128$ serving a batch of 16 requests at 16K context each: how many GB? At what context length does a single request exceed 8 GB?

<details>
<summary>Solution</summary>

```python
def kv_gb(seq_len, layers=40, kv_heads=40, d_head=128, batch=1, bytes_per=2):
    return 2 * seq_len * layers * kv_heads * d_head * batch * bytes_per / 1e9

# Batch=16, ctx=16384
print(f"Batch 16, ctx 16K: {kv_gb(16384, batch=16):.1f} GB")

# Single request exceeds 8 GB at:
for ctx in range(4096, 200000, 4096):
    if kv_gb(ctx) >= 8.0:
        print(f"Single request > 8 GB at ctx={ctx:,}")
        break
```

```output
Batch 16, ctx 16K: 163.8 GB
Single request > 8 GB at ctx=65,536
```

With 40 heads (no GQA), a 16K batch costs 164 GB — more than two A100s' worth of HBM. This is exactly why Llama-3 uses GQA (8 KV heads instead of 32).
