## The project

You will implement either a minimal SSM/Mamba block (selective scan) or an MoE FFN layer
with routing and load-balancing, pair it with a grouped-query attention (GQA) module you
write from scratch, and benchmark both throughput and quality against vanilla multi-head
attention on a real sequence task. The deliverable is a working implementation backed by
a written tradeoff analysis.

## Hard constraints (no vibe coding)

- You MAY NOT use the `mamba-ssm`, `megalodon`, `state-spaces`, or any other SSM library
  for the core SSM block. Implement the selective scan yourself in PyTorch.
- You MAY NOT use `torch.nn.MultiheadAttention` for your GQA implementation — implement
  the head sharing yourself using `torch.nn.Linear` and reshaping.
- For the MoE path: you MAY NOT use any MoE library. Implement the router (linear +
  softmax), top-k selection, and load-balancing auxiliary loss yourself.
- Benchmarks MUST use wall-clock time (`time.perf_counter`), not estimated FLOPs.
- You MAY use PyTorch's autograd; you do NOT need to write CUDA kernels.
- You MAY use standard training utilities (DataLoader, LR scheduler, AMP).

## What it must contain (and the chapter each tests)

- **SSM block or MoE FFN** (ch. 3–7): implement one of:
  - A selective-scan SSM block: input-dependent $\Delta, B, C$ projected from input,
    discretized A, recurrent forward pass (ch. 3–4).
  - A top-2 MoE FFN: linear router → softmax → top-2 selection → dispatch tokens to
    expert FFNs → weighted sum → auxiliary load-balancing loss $\alpha \sum f_i P_i$
    (ch. 6–7).
- **Grouped-query attention (GQA)** (ch. 2, ch. 1): implement attention with $n_q$ query
  heads and $n_{kv}$ key-value heads where $n_q > n_{kv}$; each K/V head is shared
  across $n_q / n_{kv}$ query heads. Verify output matches $n_q = n_{kv}$ (standard MHA)
  to floating-point tolerance. (ch. 1)
- **Throughput benchmark** (ch. 1): measure tokens/second at sequence lengths 256, 1024,
  and 4096 for (a) your alternative block and (b) vanilla MHA, on the same hardware.
  Report the ratio. (ch. 1, 2)
- **Quality benchmark** (ch. 3–12): train your model and a Transformer baseline on a
  small language modeling task (e.g., char-level Shakespeare or wikitext-2 subset) with
  equal parameter budgets. Report final validation perplexity. (ch. 9–12)
- **Needle-in-haystack or copy task** (ch. 3, 4): show whether your alternative can
  retrieve a specific token planted early in a long sequence (copy task at delay
  $T = 64, 128, 256$). Compare to a Transformer with the same depth. (ch. 3–4)
- **Tradeoff analysis** (ch. 12): apply the 6-step paper-reading framework from ch. 12
  to your own experiment — key equation, ablation (what if you remove the load-balancing
  loss? what if you use MHA instead of GQA?), and the architecture decision checklist.

## Acceptance criteria

- **SSM path**: selective scan passes the copy task at delay $T = 64$ (output token at
  position 64 matches input token at position 0 with accuracy > 95% after training).
- **MoE path**: with 8 experts and top-2 routing, the maximum expert utilization fraction
  is below $2/N = 0.25$ on the validation set (load balance achieved).
- **GQA**: output of your GQA with $n_{kv} = n_q$ (fully expanded) matches
  `torch.nn.functional.scaled_dot_product_attention` on the same inputs to `atol=1e-5`.
- **Throughput**: your implementation shows measurable advantage over vanilla MHA at
  sequence length $\geq 1024$ (at least 1.3× faster tokens/sec, or measurably smaller
  peak memory).
- **Perplexity**: your model reaches validation perplexity within 1.5 nats of a
  Transformer baseline trained with equal parameter budget and equal steps.
- **Analysis**: the tradeoff writeup explicitly addresses all four axes — active FLOPs
  per token, KV cache memory formula, training parallelizability, and recall quality.
