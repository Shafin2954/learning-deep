## Plan inference as carefully as training

A model that scores well in your notebook but can't *run* in the submission environment
scores zero. Kaggle LLM competitions are won and lost here: no internet, a time budget,
limited GPU memory, fixed hardware. Many competitors freeze a good model days early and then
spend those days just making submission *work* — do the same (F15).

## The Kaggle submission environment

- **No internet at submission.** You cannot download models, tokenizers, datasets, or pip packages
  at scoring time. Everything must be **pre-attached**: upload your model weights, adapters, and
  tokenizer as a **Kaggle Dataset**, and load from the local path. Pre-install any unusual packages
  as a dataset of wheels and `pip install --no-index` from it. Test this in a fresh notebook with
  internet off *before* you rely on it.
- **A run-time limit** (often ${\sim}9$ hours GPU for code competitions). Your inference over the
  whole test set must finish inside it — with margin. Measure throughput early.
- **Fixed GPU** (commonly T4×2 or P100). Different from your 3070 Ti — test there.
- **Reproducibility/determinism** so your selected submission is the one that scores.

## The inference-speed levers

### Merge LoRA adapters for serving

`model.merge_and_unload()` folds the adapters into the base weights → one clean model, no PEFT
overhead at inference. Save that merged model as your submission dataset. (Keep the adapters
separately for reproducibility.)

### Batched, length-sorted generation

Don't generate one example at a time. Batch the test set; sort by length before batching so each
batch pads to a similar size (less wasted attention compute — extra important with Bengali's high
token counts, ch. 3). This is often a 5–10× speedup and the difference between finishing in time
or timing out.

### Time-budget arithmetic

$$
t_{\text{needed}} = \frac{n_{\text{test}} \times t_{\text{per\_example}}}{3600} \text{ hours}
$$

must be $\le 0.7 \times t_{\text{budget}}$ (leaving a safety margin for load overhead).

```python
def will_it_finish(n_test, sec_per_example, budget_hours=9, safety=0.7):
    needed_h = n_test * sec_per_example / 3600
    usable_h = budget_hours * safety               # leave margin for load/overhead
    ok = needed_h <= usable_h
    return needed_h, usable_h, ok

for n, spe in [(5000, 0.4), (20000, 0.4), (20000, 0.08)]:
    needed, usable, ok = will_it_finish(n, spe)
    print(f"{n} examples @ {spe}s -> need {needed:.1f}h, have {usable:.1f}h usable "
          f"-> {'OK' if ok else 'TOO SLOW: batch/vLLM/shorten (this chapter)'}")
# The 0.08s row shows what batching + vLLM buys you: from impossible to comfortable.
```

```output
5000 examples @ 0.4s -> need 0.6h, have 6.3h usable -> OK
20000 examples @ 0.4s -> need 2.2h, have 6.3h usable -> OK
20000 examples @ 0.08s -> need 0.4h, have 6.3h usable -> OK
```

### Other levers

- **4-bit / quantized inference.** Load the model in 4-bit (bitsandbytes) or use a
  pre-quantized format (GPTQ/AWQ, GGUF) so weights fit and load fast.
- **vLLM** (if allowed/available): a high-throughput inference engine with paged attention and
  continuous batching. Worth it for large test sets; check it loads offline.
- **KV-cache** (F12) is on by default in `generate()` — keep it; it's what makes autoregressive
  decoding affordable.
- **Cap generation length and pick cheap decoding.** Set `max_new_tokens` tightly. Greedy or
  low-temperature (F13) is faster and usually best for classification/extractive answers.

```python
# Merge QLoRA adapters into the base, then save a clean model for offline submission.
from peft import AutoPeftModelForCausalLM
import torch

model = AutoPeftModelForCausalLM.from_pretrained(
    "out/adapter", torch_dtype=torch.bfloat16, device_map="auto")
merged = model.merge_and_unload()                 # fold adapters into base weights
merged.save_pretrained("submission_model")        # upload this folder to Kaggle
# tokenizer.save_pretrained("submission_model")   # save the tokenizer alongside!
```

```python
# Batched, length-sorted generation — finish inside the time budget.
import numpy as np
def batched_generate(model, tok, texts, batch_size=16, max_new_tokens=32):
    order = np.argsort([len(t) for t in texts])   # group similar lengths -> less padding
    outputs = [None] * len(texts)
    for s in range(0, len(order), batch_size):
        idx = order[s:s+batch_size]
        batch = [texts[i] for i in idx]
        enc = tok(batch, return_tensors="pt", padding=True, truncation=True,
                  max_length=512).to(model.device)
        gen = model.generate(**enc, max_new_tokens=max_new_tokens,
                             do_sample=False)       # greedy = fast & deterministic
        dec = tok.batch_decode(gen[:, enc["input_ids"].shape[1]:],
                              skip_special_tokens=True)
        for i, d in zip(idx, dec):
            outputs[i] = d
    return outputs
```

## The pre-submission checklist

Run it days before the deadline:

1. Fresh notebook, **internet OFF** — does everything load from your attached datasets?
2. Run inference on a *slice* — extrapolate the time; does the full set fit the budget with margin?
3. Output format **exactly** matches `sample_submission.csv` (column names, ids, order).
4. Bengali text in the output is correctly encoded (UTF-8, normalized as scored, ch. 2).
5. Re-run twice — same output? (determinism / seeds, F7).
6. Edge cases: empty inputs, very long inputs (truncation), out-of-distribution rows.
