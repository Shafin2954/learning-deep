## The highest-leverage decision

Before you train anything, make the highest-leverage decision in the competition:
*which approach does this task even need?* There are three ways to make an LLM do your
task, in increasing order of cost. Pick the cheapest one that clears your CV bar
(F14–F15). Reaching for fine-tuning first — the reflex — wastes days you don't have.

## Prompting (zero/few-shot) — cost: near zero

Use a capable instruction-tuned model as-is; describe the task in the prompt, optionally
with a few examples. No training, instant iteration. Works when: the base model already knows
the task, the label space is small/expressible in words, and you have little data. For Bengali
specifically, a strong multilingual model + good prompting (ch. 11) is a real contender, *and*
it's your fastest path to a submitted baseline (ch. 17). Limits: bounded by the base model's
Bengali ability; can't learn a domain it hasn't seen; output format can be inconsistent.

## RAG — cost: low–medium

Keep the model frozen; at inference, retrieve relevant text (ch. 12) and put it in the
prompt so the model answers *from* it. Use when the task needs **knowledge or context**
the model doesn't have. RAG adds *information*, not *skill*: it won't teach the model a
new output style or a hard classification boundary, but it will let it answer questions
about content it never memorized. Often combined with prompting.

## Fine-tuning (LoRA/QLoRA, ch. 9) — cost: medium–high

Update weights so the model *learns* the task. Use when: prompting plateaus below your
bar, you need a specific output format reliably, the task is a skill/style the base model
lacks, or you have enough labeled data to teach it (hundreds–thousands of examples). On 8 GB
it means QLoRA, not full fine-tuning (ch. 6).

## The decision procedure

Run it in order:

1. **Prompt first.** Build the zero/few-shot baseline and the embedding baseline (ch. 5)
   in week 1. Measure on your CV. This is your floor and it's nearly free.
2. **Does the task need external knowledge?** If yes → add **RAG**. If not → skip RAG.
3. **Did prompting (±RAG) clear the bar?** If yes — ship it, spend remaining time on
   data quality and ensembling (ch. 16). If no → **fine-tune**.
4. **Fine-tune the smallest viable model with QLoRA**, compare to the prompt/RAG baseline
   on the *same* CV. Keep it only if it genuinely wins beyond CV noise (F14).

```python
# A decision helper — encode the procedure so you don't skip steps under pressure.
def choose_approach(needs_external_knowledge, prompt_cv, bar, have_labeled_data):
    plan = ["1. prompt + embedding baseline (always do first)"]
    if needs_external_knowledge:
        plan.append("2. add RAG (task needs knowledge/context)")
    if prompt_cv >= bar:
        plan.append("3. baseline clears the bar -> SHIP IT; invest in data + ensemble")
        return plan
    if have_labeled_data:
        plan.append("3. baseline below bar -> QLoRA the smallest viable model (ch. 9)")
        plan.append("4. keep fine-tune only if it beats baseline on the SAME CV (F14)")
    else:
        plan.append("3. below bar but little data -> improve prompts/RAG, get data (ch.10)")
    return plan

for case in [
    dict(needs_external_knowledge=True,  prompt_cv=0.71, bar=0.75, have_labeled_data=True),
    dict(needs_external_knowledge=False, prompt_cv=0.82, bar=0.75, have_labeled_data=True),
]:
    print(case)
    for step in choose_approach(**case): print("   ", step)
    print()
```

## Mapping to task type

Classification on small data → prompting or embedding baseline often wins; if not, QLoRA a
small encoder/LLM. Knowledge QA → RAG (± light fine-tuning). Generation/translation/
summarization → fine-tune (QLoRA) a decoder or BanglaT5. Instruction/reasoning in Bengali →
prompt a strong model, then QLoRA on instruction data if it underperforms.

They combine: the strongest solutions often stack fine-tuned model + RAG + careful prompting +
ensemble. But add complexity only when the CV says the simpler thing isn't enough — never because
fine-tuning feels more "serious."

## Try it Yourself

```python
# Cost vs capability — internalize the ordering before you commit GPU hours.
approaches = [
    # name,        setup_cost, can_add_knowledge, can_learn_new_skill, iter_speed
    ("prompting",   "near-zero", False, False, "instant"),
    ("RAG",         "low-med",   True,  False, "fast"),
    ("QLoRA FT",    "med-high",  False, True,  "slow (GPU)"),
    ("FT + RAG",    "high",      True,  True,  "slowest"),
]
print(f"{'approach':12s}{'setup':11s}{'+knowledge':12s}{'+skill':9s}{'iteration'}")
for n, c, k, s, it in approaches:
    print(f"{n:12s}{c:11s}{str(k):12s}{str(s):9s}{it}")
# Read it top-down: start cheap, climb only when your CV forces you to.
```

